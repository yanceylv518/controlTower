# v2.7-B3 稀疏指标曲线修正交付说明

## 交付内容

1. `TrendSeries` 新增可选 `sparse` 标志。稀疏序列启用 `connectNulls` 与小尺寸 symbol，跨空分钟仍能连线，单个孤立点也可见；未标记的请求量、错误率等密集序列保持原渲染行为。
2. TTFT 平均、TTFT P95 与缓存命中率标记为稀疏序列。TTFT 图明确提示“无流式流量的时段无数据”，不把缺失值画成 0。
3. 1m→5m 汇总将非空 `ttft_p95_ms` 取最大值；全部为空时仍为空。该值是合并集 P95 的保守上界近似，不等同于原始样本重算的精确 P95。
4. MySQL 5m 增量合并和内存存储使用同样的 MAX 规则，避免后续批次覆盖或清空已生成的近似值。P50/P99 与 use_time 的直方图逻辑未改变。
5. 修复 1m 根因：同一分钟内多次 Agent 部分聚合到达时，`MergeMetric` 不再把 `ttft_p95_ms` 清空，而是对两侧非空值取 MAX；一侧为空保留另一侧，双空保持空。

## 缓存命中率核对结论

`big_input_count` 和 `big_input_cache_hits` 在 5m rollup 中按可空计数器分别相加，Dashboard 读侧使用两者计算 `cache_hit_rate`，因此 5m/6h/24h 视图已能正确输出缓存命中率。新增聚合测试同时覆盖跨分钟求和结果。

## 分位数现状说明

Agent 当前每 30 秒上报一次，同一个 1m 桶可能由多个部分聚合合并。TTFT P95 本批使用部分 P95 的 MAX 作为保守上界；use_time 的 P50/P95/P99 在发生桶内合并后由合并直方图插值，不能再表述为原始样本的“1m 精确分位数”。彻底精确需要 Agent 跨 pass 持有开放分钟桶并在关桶后一次上报，作为后续 Agent 升级候选，本批不修改采集语义。

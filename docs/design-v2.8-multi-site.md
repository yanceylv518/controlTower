# 设计 v2.8：多站点/多实例模型（站点显式化）

日期：2026-07-23。背景：pinducloud_cn 准备上阿里云 ALB 负载均衡（两台服务器各跑一个 newapi 实例），后续 pinducloud_h 也可能同样扩展。CT 需要从"实例=站点"的隐含一一对应，升级为显式的"站点 → 多实例"两层模型。

## 1. 概念模型

```
站点（site）           实例（instance/机器）           agent
pinducloud_cn   ──┬──  pinducloud_cn_1（A机，业务采集）←── agent A
                  └──  pinducloud_cn_2（B机）          ←── agent B
pinducloud_h    ────   pinducloud_h_1（现状单机）       ←── agent
```

- **站点** = 一套 newapi 业务：一个共享 MySQL、一个入口（ALB 或单机直连）。
- **实例** = 一台服务器上的一个 newapi 进程；每台机器一个 agent，InstanceID 即实例 ID。
- 命名约定：实例 ID = `<站点>_<序号>`（如 `pinducloud_cn_1`）；site_id 填站点名。

## 2. 数据归属分界（核心原则）

**能不能分实例展示，取决于数据源里有没有实例字段。** 这是数据源决定的，不是设计选择。

| 类别 | 数据 | 来源 | 归属 |
|---|---|---|---|
| 站点级 | 业务指标（流量/错误/延迟/token/缓存）、日志样本 | 共享库 logs 表 | 全站一份事实，**每站点仅一个 agent 采集**（游标机制决定，双采即翻倍）。约定由 1 号机 agent 承担；上报挂采集实例名下，语义属于站点 |
| 站点级 | 渠道快照 | 共享库 channels 表 | 同上，仅采集 agent 开启 |
| 实例级 | 资源指标、docker 状态、健康探测、心跳在线、nginx 分诊 | agent 采本机 | 每 agent 各报各的，天然分实例 |

推论：newapi logs 表没有"哪个实例处理"字段，**分实例业务量不存在**，任何页面都不应假装能展示它；ALB 怎么分流 CT 看不见（数据源天花板，记档）。

## 3. 页面语义

| 页面 | 逻辑 |
|---|---|
| 总览 | 顶部站点切换器，**一次只看一个站点**（用户拍板）；数据=该站点业务指标 |
| 渠道/模型/客户维度页 | 按站点筛选（渠道/客户本就是站点属性） |
| 系统状态页 | 站点做分组标题，实例做行（资源/容器/健康/在线） |
| 实例维度页 | 业务指标区标注"全站汇总"（挂采集实例名下），不假装分实例 |
| 告警通知 | 文案带站点名+实例名（如"pinducloud_cn 节点 2 离线"） |

## 4. 数据模型改动

- instances 表加 `site_id VARCHAR(64) NOT NULL DEFAULT ''`（014 迁移，纯增量 ADD COLUMN，1060 容忍幂等，禁止表重建型 ALTER——010 教训）。
- **读时回退**：site_id 为空 → 视为等于 instance_id（"每实例自成一站"）。存量数据零迁移、升级当天页面不变；单实例站点是本模型的退化情形而非特例。
- 指标表**不动**：业务指标仍按 instance_id 存，站点视图=站点成员实例集合的 IN 过滤/汇总（复用 008 索引与既有 instance_id 下推路径）。采集角色未来若换机，历史数据靠集合过滤仍完整。

## 5. 批次拆分与部署顺序

| 批次 | 内容 | 依赖 |
|---|---|---|
| **v2.8-B1（agent）** | `CT_LOG_COLLECT_ENABLED` 开关：关闭后不碰共享库，只跑心跳/健康/资源/docker/nginx——B 机 agent 的部署前提 | 无 |
| **v2.8-B2（server 数据层）** | 014 迁移、实例管理站点字段、系统状态页按站点分组、离线/告警文案带站点 | 无（与 B1 并行） |
| **v2.8-B3（server 业务视图）** | 总览站点切换器、维度页站点筛选（API 增量 `site` 参数）、实例维度页语义标注 | B2 |

部署顺序：B1 出 agent 包 → B 机部署 agent（此时页面多出一个未分组实例，可接受）→ Server 先 B2（014 自动迁移）后 B3 → 打 tag。B2/B3 与在途 v2.3-B2 web 批次可能撞 web 文件，冲突 rebase 手工合并。

## 6. 部署侧清单（不改代码，上 ALB 时执行）

1. **ALB 健康检查指向 `/api/status`**——流量层摘除与 CT 探活是两回事：ALB 管"坏了别导流量"，CT 管"坏了通知你"，两个都要有。
2. **nginx 去留待定**：延迟分诊（v2.2-B1）依赖 nginx access log。ALB 直连 newapi 则分诊数据源断；每机保留 nginx 则 B 机 agent 配 `CT_NGINX_ACCESS_LOG` 补全（nginx 采集按 agent 走，不涉共享库，两份不冲突）。注意 ALB 健康检查高频日志勿污染延迟统计。
3. **两机 NTP 对时**（分钟桶聚合依赖）。
4. **B 机 → CT server 网络连通性**预先验证。
5. B 机 agent 配置要点：`CT_LOG_COLLECT_ENABLED=false`、不配 LogDSN/企微 webhook、ChannelSnapshot 关、InstanceID=`pinducloud_cn_2`、探活指本机 `127.0.0.1:3000/api/status`。渠道控制保持 A 机指固定实例（勿指 ALB 入口，避免摘除时控制通道随之不可用）。

## 7. 明确不做

- 分实例业务量展示（数据源无此字段）。
- 独立 sites 表 / 站点 display_name（site_id 字符串直接展示，需求出现再议）。
- 多站点告警路由分群（现阶段同一企微群）。
- 整站流量在 ALB 层的观测（属于云监控范畴）。

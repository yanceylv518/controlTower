# v2.3-B3 维度页加载性能优化交付说明

日期：2026-07-15

## 实现

- 新增迁移 `008_metric_indexes.sql`，为 `metric_1m`、`metric_5m` 增加 `(dimension_type, instance_id, dimension_key, bucket_time)` 复合索引；保留原 `bucket_time` 前导索引供 retention 使用。
- latest 查询由逐行相关子查询改为“24 小时活跃集合分组 + 自联结”，`dimension_type` 在 SQL 内下推，API 响应字段不变。
- 语义变化：超过 24 小时没有指标的维度不再出现在 latest 列表；历史查询不受影响。
- 内存存储同步 24 小时与维度类型语义，Overview 只查询 `instance` 维度。
- 渠道名、用户名改为按实例整批预载，60 秒内单键解析只查内存；100 个键测试确认每种类型只进行一次批量查询。
- 维度页 metrics/快照返回后立即渲染列表与指标卡，历史曲线在独立 loading 区域异步加载。
- Dashboard JSON 响应支持按 `Accept-Encoding` 协商 gzip，并通过 `sync.Pool` 复用 writer；Agent heartbeat/report 路由未接入该中间件。
- `deploy/perf-seed.sql` 可在一次性测试库生成 1,209,600 行数据，并附 latest 与 history 的 EXPLAIN 语句。

## EXPLAIN 对比

旧查询形态为 `DEPENDENT SUBQUERY`：外层每个候选行执行一次 `MAX(candidate.bucket_time)`，旧索引以 `bucket_time` 开头，无法有效服务 instance/dimension 相关条件。

新查询预期计划：派生表按 `idx_metric_1m_dim_bucket` 扫描 24 小时活跃维度并分组，外层通过主键/复合键回表；history 查询以 `dimension_type`、`dimension_key` 和时间范围使用新索引。

当前开发机 MySQL 可用但没有测试库管理员凭据，无法安全灌入 120 万行，因此未伪造实测耗时或 EXPLAIN 输出。部署到一次性基准库后执行：

```bash
mysql -u<benchmark-user> -p control_tower < deploy/perf-seed.sql
```

目标：latest 查询热缓存与冷缓存均记录，百万行级目标 `<100ms`。禁止在生产库直接运行造数脚本。

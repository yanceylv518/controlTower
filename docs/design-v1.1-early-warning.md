# v1.1 预警设计：消除 600 秒盲区与提前预警

状态：设计稿（未实现）。对应 `iteration-log.md` v1.0 遗留问题与「600 秒超时盲区」分析。

## 1. 问题定义

v1.0 的告警只基于 new-api `logs` 表中**已完成**的请求：

1. **盲区**：一个挂到 600 秒才写入 504 的请求，告警最快也在 600+30 秒后；执行中的请求完全不可见。渠道整体挂起时，恰恰是"最需要告警的时刻没有任何新日志产生"。
2. **滞后**：等到窗口里攒够 3 条错误，客户已经失败了 3 次。没有任何"事情正在变坏"的领先指标。

边界约束不变：不修改 new-api 源码/路由/Nginx 配置、不进入用户请求链路、不读请求体/响应体/密钥。

## 2. 设计原则

- **信号分层**：不同信号回答不同问题，互相补位，而不是堆一个万能规则。
- **由被动到主动**：v1.0 是纯被动（等日志）；v1.1 引入主动信号（探测）和负向信号（静默）。
- **零新增依赖优先**：优先用 Agent 已有的数据面（logs 表、admin API 凭证），可选模块才引入新数据源（Nginx 日志）。
- **复用 v1.0 的通知骨架**：episode 去重、恢复重臂、失败重试、钉钉发送全部复用，只新增规则类型和消息模板。

## 3. 四类信号设计

### 信号 A：主动渠道探测（提前预警的主力）

**思路**：不等客户流量，Agent 周期性地主动验证每个启用渠道是否可用。渠道刚开始挂起时，探测在 1~2 个周期内就会失败——**在客户大量踩坑之前告警**，同时天然覆盖 600 秒盲区（探测自己带短超时）。

**实现路径**：复用 `channelcontrol` 已有的 new-api 管理端凭证与登录逻辑，调用 new-api 自带的渠道测试接口（`POST /api/channel/test/{id}`，可带 model 参数）。它通过 new-api 真实打到上游，返回成败与延迟，且能精确定位到渠道——比自建探测请求干净，也不进用户请求链路。

**规则**：
- 探测失败/超时后**不等下个周期，15 秒后立即复核一次**；复核仍失败（连续 `CT_PROBE_FAILURE_THRESHOLD`，默认 2）→ 告警「渠道探测失败」。立即复核把瞬间挂起的最坏发现时间从 ~4.5 分钟压到 **~2.5 分钟（平均 ~1.5 分钟）**，已接近"120 秒算挂起"定义的理论下限；日常成本不变，仅疑似故障时多一条测试消息。
- 探测延迟连续 N 次超过阈值 → 预警「渠道响应变慢」（领先指标）。
- 探测恢复成功 → 供信号 D 恢复判定计数。
- **跨信号升级**：其他信号出现可疑（静默线索、慢返回窗口过半、探测耗时突增）时，对该渠道提前加急探测，不等例行周期——信号互相点火，而非各等各的阈值。

**证据驱动，不定时必打（核心省钱设计）**：每到探测时点先查日志——该渠道最近 `CT_PROBE_FRESHNESS_SECONDS`（默认 120）秒内有成功完成的客户请求，则**跳过本次探测**（流量本身就是健康证明，无需花钱重复验证）；没有新鲜证据才真正发探测。效果：

- 忙渠道几乎 0 条探测/天（客户流量持续供证）；闲渠道只在空闲时段探测，空闲间隔可放宽到 `CT_PROBE_IDLE_INTERVAL_SECONDS`（默认 300，无人使用时晚发现几分钟无实际伤害）。总量对比"定时必打"降 80~95%。
- **检测速度不受损**：渠道挂起 = 成功完成停止 = 新鲜证据 2 分钟内自然过期 → 探测自动恢复发射 → 失败 + 立即复核 → 告警（~2.5-3 分钟）。挂起本身会自动唤醒探测——探测频率自动跟随信息缺口。
- 与信号 B 逻辑统一：静默线索（3 分钟无日志→加急探测）与证据驱动例行探测（2 分钟无成功→例行探测）是同一机制的两档灵敏度。

**成本与安全**：
- 单条测试消息约 10~20 token，`CT_PROBE_MODEL` 指定廉价模型时单条约 0.00001 元级；证据驱动后忙渠道日成本≈0，闲渠道最坏约 288 条/天（仅空闲时段）。
- 探测超时 15 秒；渠道多时错峰打散，避免同时打爆上游。
- 需要 admin 凭证，**默认关闭**（`CT_PROBE_ENABLED=false`），显式开启才生效。
- 只探测启用状态的渠道；渠道列表来自已有的 channelcollector 快照查询。

**检测时延**：渠道挂起后 ≤ 探测间隔 + 超时 + 立即复核 ≈ 最坏 **2.5 分钟、平均 1.5 分钟**（对比 v1.0 的 10+ 分钟）。

### 信号 B：完成静默检测（用已有数据破盲区）

**思路**：请求挂起的直接后果是"该维度不再产生新的完成日志"。**沉默本身就是信号**。不需要任何新数据源，从现有 logs 采集流里就能算。

**规则**：对每个近期活跃的渠道（以及实例整体）：
- 基线：过去 `CT_SILENCE_BASELINE_MINUTES`（默认 15）分钟内完成数 ≥ `CT_SILENCE_MIN_BASELINE`（默认 5）条，才认为该维度"应该有流量"。
- 触发：满足基线的维度连续 `CT_SILENCE_MINUTES`（默认 3）分钟完成数为 0 → 产生静默线索（处理方式见下）。
- 实例级：整个 logs 表停止增长（而此前活跃）→ 升级告警「实例疑似整体挂起」。

**定位澄清：静默是线索，不是证据**。静默检测从"证据缺失"做推断，天然区分不了"请求都挂住了"和"流量正常地停了"（大客户下班、定时任务结束、客户自行切走）——两者在日志里完全同形。因此静默条件满足时**不直接告警，而是触发验证**：

```
静默条件满足（渠道连续 3 分钟无日志且其他渠道正常）
  → 立即对该渠道发起一次即时探测（15 秒超时，不等例行探测周期）
  → 探测失败：【告警】渠道挂起（静默 + 探测双重确认，非猜测）
  → 探测成功：不告警（渠道通，只是没人用），仅记内部日志
```

静默窗口取 3 分钟（此前草案 5 分钟）：因为静默只触发探测而非直接告警，误报代价只是多一次探测，可以激进——无探测前奏的瞬间挂起约 **3.5 分钟**内完成双重确认。

**分级扩展：吞吐骤降线索（覆盖部分挂起）**。部分挂起（如 50% 请求挂住、其余正常）时渠道不会静默——成功请求持续出日志。因此在"零完成"之外增加一档：

- 渠道完成速率跌到自身基线（过去 15 分钟均值）的 `CT_THROUGHPUT_DROP_RATIO`（默认 50%）以下、持续 3 分钟、且实例整体正常 → 吞吐骤降线索 → 对该渠道**连发 `CT_BURST_PROBE_COUNT`（默认 3）次突发探测**（间隔数秒）——部分挂起时单次探测可能碰巧成功，多连发中任一次超时即证明有请求无法返回 → 【告警】渠道部分挂起。
- 吞吐骤降状态下，证据驱动探测的"有成功完成即跳过"逻辑**失效**（部分挂起时成功完成不构成健康证明）。

**覆盖边界（诚实声明）**：吞吐骤降线索对重度部分挂起（≥50%）有效；轻度部分挂起（20~30% 请求挂）吞吐降幅淹没在自然波动里、突发探测命中概率也有限，当前边界内**没有可靠的早期检测手段**，只能等 600 秒后 504 写入日志由 v1.0 错误窗口触发（~10 分钟）。轻度部分挂起的早期精确检测是 v1.2 中间件方案（请求开始事件 → 在途计数）不可替代的核心场景：任何比例的挂起都在 ~2.5 分钟内可数、可定位到客户与渠道。

未启用探测（未提供 admin 凭证）的部署：静默只发【预警】级消息，措辞"疑似"，且可通过 `CT_SILENCE_ENABLED=false` 整体关闭。

**误报控制**：
- 基线条件排除了本来就没流量的冷渠道。
- 夜间自然低谷：实例整体静默用更长窗口（默认 10 分钟）+ 可配置静默时段；渠道级静默要求"实例整体仍活跃而该渠道独自沉默"。
- 最终误报由"静默→探测确认"链路兜底：猜测只决定何时验证，结论由探测给出。

**检测时延**：挂起后约 **3~3.5 分钟**发现线索 + 探测 15 秒确认，零成本、零新增权限（确认步骤需探测凭证）。

### 信号 C：趋势恶化预警（领先指标）

**思路**：超时故障通常有前兆——延迟先爬升，然后才雪崩。对已采集的事件做跨窗口趋势判断：

- 渠道/实例的 use_time P95 连续 3 个评估窗口逐窗上升且累计涨幅 >50% → 预警「延迟持续恶化」。
- 错误率跨窗口爬升（如 0% → 10% → 20%）但尚未触发 v1.0 阈值 → 预警「错误率上升趋势」。

这类消息标记为**预警级**（区别于告警级），钉钉消息前缀区分【预警】/【告警】，让人知道"还没坏，但在变坏"。

### 信号 F：慢返回窗口规则（已完成请求的耗时告警）

**思路**：`logs` 表每条记录带 `use_time`（请求总耗时，秒）。600 秒超时雪崩之前通常先出现一批 120~300 秒才返回的慢请求——慢返回是超时的前奏。对它设窗口规则，能比等超时提前数分钟发现恶化。

**规则**（与 v1.0 错误窗口完全同构，同一套窗口/episode/去重机制，窗口里多数一个指标）：
- 按渠道/客户维度，最近 `CT_ALERT_SLOW_WINDOW`（默认 10）条请求中，`use_time ≥ CT_ALERT_SLOW_SECONDS`（默认 120）的达到 `CT_ALERT_SLOW_THRESHOLD`（默认 3）条 → 【告警】渠道/客户慢返回激增。

**误报控制**：流式长回答（生成大量内容的 stream 请求）正常也可能超过 120 秒。提供 `CT_ALERT_SLOW_STREAM_SECONDS`（默认 300）给流式请求单独的更高阈值，或设为 0 排除流式请求。

**能与不能**：此规则识别的是"**返回了但很慢**"，检测时点在请求完成时。"**在途未返回**"的精确识别需要请求开始事件，数据面没有（new-api 只在请求结束时写日志），改 new-api/网关越界；TCP 连接年龄跟踪可近似但 keepalive 空闲连接与合法长流式连接无法区分，误报率高，不采用——"在途挂起"场景由信号 A（探测）和信号 B（静默）覆盖。

### 信号 D：恢复通知（闭环体验）

**原则：恢复必须基于正向证据（观察到足量成功），"触发条件不再满足"不构成恢复。** 否则会出现：窗口错误降到 2/10（仍 20% 错误率）就报恢复、阈值附近抖动导致告警/恢复交替刷屏、以及最严重的——静默告警在 600 秒超时 504 洪峰涌入时因为"重新有日志了"而误报恢复（恰恰是故障最重的时刻）。

各告警类型的恢复判定：

| 告警类型 | 恢复条件 |
| --- | --- |
| 错误窗口 / 慢返回 | 告警后新完成 ≥10 条请求（窗口彻底换血），其中失败/慢返回 ≤1 条；旧错误单纯滑出窗口不算 |
| 探测失败 | 连续 3 次探测成功（约 6 分钟持续正常） |
| 静默（挂起确认） | 出现 ≥10 条新的**成功**完成；超时错误洪峰不算恢复——静默结束伴随错误洪峰时不发恢复，转交错误窗口告警 |

通用防抖：恢复条件满足后再观察一个评估周期无恶化才发送；同一 episode 恢复消息只发一次。

**episode 的三种收尾（人工处置感知）**。真实运维中告警渠道常被人工禁用，流量归零后"足量成功"永远等不到，episode 会变僵尸。利用已有的渠道快照状态数据（channelcollector），给 episode 定义完整生命周期：

```
告警中的渠道
├─ 观察到足量成功           → 【恢复】（上游自愈，LLM 上游常见）
├─ 快照发现渠道被禁用       → 【已处置】渠道已被人工禁用，episode 立即闭环
└─ 此后快照发现重新启用     → 立即加急探测验证：
                              通过 → 【重新上线】启用且探测验证通过
                              失败 → 【告警】重新启用但探测失败（防带病上线）
```

- 每条告警必有确定结局（恢复/已处置/重新告警），无僵尸 episode；兜底：`CT_EPISODE_MAX_HOURS`（默认 24）超时自动关闭，不发消息。
- 渠道状态检测需只读账号追加 `GRANT SELECT ON newapi.channels`（部署文档注明）；未授权的部署退化为"仅正向证据恢复 + 超时关闭"。
- "重新启用即验证"同时覆盖一个原设计未考虑的风险：把未修复的渠道放回线上。

**成本说明：恢复判定不产生额外请求。**"新完成 ≥10 条"统计的是客户自然流量的完成记录（旁观 logs 表）；"连续 3 次探测成功"是对**既有例行探测节奏**的计数，不为恢复加发探测。整个 v1.1 唯一的额外主动请求是静默线索触发的单次加急探测（每次故障事件一条）。持续成本只来自例行探测本身，且探测是证据驱动的（忙渠道自动跳过，见信号 A）：忙渠道日成本≈0，闲渠道最坏约 288 条/天、单条约 0.00001 元级。

消息措辞陈述事实、不下结论（与静默检测"线索→验证"同一哲学：报证据，不替人判断）：

```
【恢复】渠道 18
告警后新完成 12 条请求，0 失败，探测连续 3 次通过
故障持续约 18 分钟（14:03 告警 → 14:21）
```

配套配置：`CT_RECOVERY_MIN_SUCCESS`（默认 10）、`CT_RECOVERY_MAX_FAILURES`（默认 1）、`CT_RECOVERY_PROBE_SUCCESSES`（默认 3）。

### 信号 E：Nginx timing 日志（2026-07-13 二次决策：移出告警体系，走监控分析线）

生产两台 new-api 的 Nginx 已配置 `timed` 日志格式（rt/uct/uht/urt/bytes，见 `latency-diagnosis.md`）。**用户决策：这路数据不发钉钉、不产生告警消息**——它是分析型数据，在 Control Tower Web 里看：Agent 只读 tail → 分钟桶聚合（TTFT/传输段分位数、5xx/504 计数、慢请求归因计数）→ 随既有上报链路入库 → Web「延时分诊」页展示趋势与归因。实现批次：`codex-task-v2.2-b1-nginx-timing-analytics.md`。

- 局限不变：识别不了渠道/客户（业务维度仍由 logs 采集负责）。
- **失效安全（硬性要求）**：`CT_NGINX_ACCESS_LOG` 留空 → 模块完全不启动，零副作用；配置了但文件不存在/无读权限/被 logrotate 轮转走 → 打一条 WARN 后继续运行、周期性重试打开，**绝不 panic、不退出进程、不影响其他信号**；日志行不含 timed 字段（旧格式/其他 vhost 混写）只静默跳过该行，不告警不刷屏。与 v1.0 事件日志的 fail-safe 同一原则：可选模块的任何故障都不能伤害主链路。
- 覆盖矩阵中 E 列的"告警"能力据此作废；将来若确需网关侧即时告警（如 504），基于已入库的 timing 数据另起批次，不在 Agent 端发消息。

配套增强（仍归挂起的探测批次）：**网关开销分解探测**——经 new-api 的整链探测耗时，减去不带 key 的上游 TCP/TLS 握手基线探测（不读渠道密钥，不违边界），得到"new-api 内部开销"曲线——直接回答"延时大是不是 new-api 的问题"。

| 场景 | v1.0 窗口规则 | A 探测 | B 静默 | F 慢返回 | C 趋势 | E 网关日志 |

| 渠道快速返回错误 | ✅ ~1 分钟 | ✅ | — | — | 前兆可见 | ✅ |

结论：A+B 组合消除盲区并覆盖"无流量时坏掉"这一 v1.0 完全无解的场景；F 抓住超时的前奏（慢返回）；C+D 提供提前量和闭环；E 锦上添花。

## 5. 版本切分建议

- **v1.1**：信号 B（静默检测）+ 信号 F（慢返回窗口规则）+ 信号 D（恢复通知）+ 信号 A（主动探测，默认关闭但**强烈建议开启**）。理由：B/F/D 零新增权限、无成本、与现有窗口机制同构；A 是整套体系的快速反应主力——瞬间全挂场景的 1.5~2.5 分钟发现时间完全依赖它，不开探测时只有静默的 ~3.5 分钟预警级提示。部署文档需引导用户算成本账后开启。
- **v1.2**：在途请求检测（new-api 轻量中间件产生请求开始事件，Agent 配对完成记录——核心场景是轻度部分挂起的早期精确检测，v1.1 所有信号对此均无可靠手段；需用户批准打破"不修改 new-api"边界）+ 信号 C（趋势预警）+ 信号 E（Nginx 模块）+ 钉钉加签。
- 顺路必做：v1.0 遗留的 logs 采集 NULL 防护（P0，同样影响独立模式采集稳定性）。

## 7. 消息分级与模板

- 【告警】红色语义：探测连续失败、静默+探测双重确认的渠道挂起、v1.0 错误窗口触发、慢返回窗口触发、实例整体静默。
- 【预警】黄色语义：延迟恶化趋势、错误率爬升、探测变慢、未经探测确认的静默（无 admin 凭证部署）。
- 【恢复】绿色语义：任一维度回到健康。
- 模板延续 v1.0：标题 + 实例 + 维度标签 + 量化描述 + 时间；探测类附最近一次失败原因摘要。

## 8. 信号持久化与可视化（为 Web 可见性打地基）

2026-07-13 Web review（`review-web-monitoring-2026-07-13.md` P1-2）发现：Agent 独立模式的真实告警（episode 触发/提醒/恢复）只存在于内存和钉钉群，Web 完全不可见。v1.1 实现时补齐地基：

- **本地持久化**：每个 episode 事件（alert / remind / recover / disposed / probe_fail / silence_confirmed）追加写 `CT_DATA_DIR/alert-events.jsonl`（JSON lines，按大小轮转保留最近 2 个文件），字段：时间、维度、kind、窗口统计、episode 起始与累计错误。价值：重启后可追溯、故障调查不再依赖钉钉群翻历史（渠道 26 事故的调查成本教训）。
- **上报扩展**：双模式（配置了 `CT_SERVER_URL`）下，report 增加 `alert_events` 数组增量上报，Server 落库（v2.0 前 Server 侧仅存储）。
- **可视化**：v2.0 Web 增加"Agent 告警时间线"（按维度的 episode 列表与事件流），与 Server 端规则告警在告警中心并列展示、来源标注。

## 9. 测试与验收要点

- 静默检测：构造"有基线后停止产生日志"的事件序列单测；夜间低谷不误报（无基线不触发）；实例活跃但单渠道静默触发渠道级告警。
- 探测：httptest 模拟 new-api 测试接口的成功/失败/超时/慢响应；连续失败阈值与恢复重臂；错峰调度。
- 恢复通知：每种告警类型触发→恢复→收到恢复消息且不重复。
- E2E：真实环境验收方式——手动禁用一个渠道的上游 key，观察探测告警时延；用 iptables 丢包模拟挂起，观察静默告警时延。

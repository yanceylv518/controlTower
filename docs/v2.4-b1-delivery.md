# v2.4-B1 Request ID 关联延时诊断交付说明

## 交付结果

本批次已打通 Nginx 慢样本到 Control Tower 使用日志维度的精确关联链路：Agent 从 timed access log 解析 `request_id`，随慢样本上报；Server 按 `(instance_id, request_id)` 批量查询 `log_samples` 与 `log_events`，Web 在延时分诊页展示用户、渠道、模型、令牌和关联状态。

关联只使用精确 Request ID，不使用时间邻近推测。一个 ID 命中一条业务日志时为 `matched`，无命中为 `unmatched`，去重后仍命中多条时为 `multiple`，页面不会从多条记录中任意选择维度。

## 数据链路

1. Nginx timed access log 输出 `request_id=$upstream_http_x_oneapi_request_id`。
2. Agent 解析 `request_id`；缺失、`-` 或超过 255 字节时仅将 ID 置空，不丢弃整条 timing 日志。
3. 慢样本的聚合、Top 5 截断及幂等键保持不变，新增字段通过 Agent report 上报。
4. 迁移 `009_nginx_sample_request_id.sql` 为 `nginx_slow_samples` 增加可空字段和 `(instance_id, request_id)` 索引；迁移不包含启动时重建表的语句。
5. Dashboard 一次批量读取当前页全部 Request ID，先合并 `log_samples`，再补充 `log_events`，相同 `source_log_id` 去重，避免 N+1 查询。
6. `/api/dashboard/nginx-timing/slow-samples` 支持 `user_id`、`channel_id`、`model_name`、`match_status` 精确筛选。

## 页面行为

- 慢样本表新增用户、渠道、模型、令牌、Request ID、关联状态和复制操作。
- 已关联维度可跳转到对应监控页；`multiple` 明确显示“多条重试/尝试”，不展示伪造维度。
- 页面明确区分 RT（请求总耗时）、UHT（上游首响应时间）和 URT（上游总耗时）；UHT 不宣称为模型精确 TTFT。
- 聚合趋势仍为实例级 Nginx 指标，不因维度筛选改变。

## 兼容性与限制

- 旧 Agent、旧 payload 和历史慢样本没有 Request ID，均安全显示为 `unmatched`。
- `aggregate_with_samples` 下业务关联主源是 `log_samples`；`full_debug` 下 `log_events` 可补充更多记录。
- `log_samples` 有采样上限。错误风暴或截断时，Nginx 慢样本可能找不到对应业务样本，这是设计内的 `unmatched`，不会降级为时间猜测。
- 本批次不修改 new-api，不影响独立错误告警、企业微信通知、渠道控制或原有日志采集。

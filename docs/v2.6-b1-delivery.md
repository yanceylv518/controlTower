# v2.6-B1 可靠性三件套交付说明

## 交付内容

1. Agent full pass 调整为“读取日志与本地企业微信告警 → Server 补传、心跳、命令与上报”。Control Tower Server 的补传、心跳或上报失败时，本轮已采集事件仍进入本地缓冲，日志游标照常推进，本地告警不再被 Server 故障阻断。
2. Server 新增 `instance_offline` 合成告警。仅启用且曾经收到心跳的实例参与判断；离线阈值读取 settings provider 的 `CT_OFFLINE_ALERT_SECONDS`，默认 120 秒；心跳早于 7 天的实例视为退役，不再持续 firing。
3. 通知渠道新增 `wecom`。企业微信机器人收到带 `[告警]` 前缀的文本消息；HTTP 2xx 且响应 `errcode=0` 才算成功，失败沿用既有指数退避、耗尽和手动重发机制。

## 两条独立通知链路

- Agent 直发：读取 new-api 数据库后直接向企业微信群机器人发送近期错误提醒，不依赖 Control Tower Server。
- Server 通知中心：离线、资源、错误率和 P95 等合成告警通过通知渠道发送，可选择企业微信、钉钉或通用 Webhook。

建议两条链路使用不同群，或至少通过机器人名称和消息前缀区分，避免值班人员把“业务请求错误”和“监控平台合成告警”混为一类。

## 行为说明

心跳返回的 `ServerLastLogID` 从本轮采集前对齐改为下一轮生效，因此 Server 恢复后的补传游标可能晚一轮对齐。正常运行不受影响；数据库唯一键与幂等写入会吸收这一个周期内可能产生的重复事件。

- `deploy/e2e-server.sh` 的失败通知渠道已改为 `wecom`，继续覆盖失败投递与手动重发路径。
- 单测覆盖三类 Server 故障的缓冲/游标行为、离线触发/从未接入/恢复/退役分支，以及企业微信成功、非零 errcode 和重试次数。

生产手工验证需在 Linux 环境分别执行：让 Agent 指向不可达 Server 确认本地告警继续；停 Agent 超过配置阈值确认离线告警与企微推送；恢复 Agent 确认自动 resolve。

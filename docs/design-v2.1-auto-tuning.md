# v2.1 设计：渠道质量自动调权（优先级/权重）

状态：设计稿。排期：v2.0 上线观察期结束后启动。对应原规划 P9（自动调权默认关闭、建议可解释、动作有审计）。

## 1. 需求与设计哲学

按渠道质量自动调整其在 new-api 中的优先级与权重。用户要求：①质量判断标准可配置；②自动可开关；③**关闭自动时也要产出数据，用于判断"开自动是否合理"**。

设计哲学延续项目一贯原则：**猜测只触发验证、结论靠正向证据、消息报事实不替人判断**——放到调权场景就是：引擎先当"只动嘴的顾问"（影子模式），每个建议带完整证据快照并事后回看对错；人先用"一键采纳"半自动用一阵；命中率数据说服你之后，才把方向盘交给它——而且交了也留着随时收回和一键回滚。

## 2. 三档模式（核心机制，逐档递进）

| 模式 | 引擎行为 | 用途 |
| --- | --- | --- |
| **observe 观察（默认）** | 评估→生成建议→**只记录不执行**；30 分钟后自动回填"事后走势"（若当时执行了会怎样的对照证据） | 积累决策记录与命中率，回答"开自动合不合理" |
| **confirm 建议+人工确认** | 建议推到 Web（可选钉钉提醒），人工点"采纳"→ 走既有渠道命令链路（confirm 语义、Agent 执行、审计落库） | 半自动过渡期，人机协同 |
| **auto 自动** | 引擎直接创建渠道命令执行（created_by=`auto-tuner`），全程护栏约束 + 每次可一键回滚 | 数据证明可信后的终态 |

模式按实例配置，随时切换；**任何模式下建议记录与事后回看都在积累**——auto 模式的执行记录同样回填效果，持续自证。

## 3. 质量评估与可配置标准

**数据源**（全部现成）：`metric_1m` 的 `instance_channel` 维度（错误率/请求数/P95/慢占比）+ 渠道快照（当前权重/优先级/状态）+ 探测结果（v1.1 上线后可选并入）。

**策略配置**（每实例一份，Web 可改，存 CT 库）：

```
evaluation_window_minutes: 15      # 评估窗口
min_samples: 20                    # 窗口内请求数低于此不评估（小样本噪声防护）
degrade:                           # 降级规则
  error_rate_threshold: 0.15       # 错误率 ≥15%
  sustained_windows: 2             # 连续 2 个评估周期满足才触发（防抖）
  weight_step_ratio: 0.5           # 建议权重 = 当前 × 0.5（向下取整，不低于 floor）
  weight_floor: 1                  # 权重下限（不建议降到 0——由熔断规则单独处理）
severe:                            # 熔断规则（可选启用）
  error_rate_threshold: 0.5        # 错误率 ≥50% 持续 1 周期
  action: priority_drop            # 降优先级（比清权重温和，保留兜底路由）
recover:                           # 恢复规则（只对"被本系统降过"的渠道生效）
  error_rate_threshold: 0.02       # 错误率 ≤2%
  sustained_windows: 4             # 持续 4 个周期
  weight_step_ratio: 2.0           # 阶梯回升（每次翻倍，封顶原始值）
cooldown_minutes: 10               # 同一渠道两次动作最小间隔
```

规则刻意保持**阈值型而非评分型**——每条建议都能用一句话解释（"错误率 18% 持续 2 个周期 ≥ 阈值 15%"），符合"建议可解释"的验收要求；将来真需要综合评分再演进。

## 4. 护栏（auto 模式的安全边界，全部硬编码兜底 + 部分可配）

1. **最小可用集**：一次评估中被降级的渠道数不得使"健康渠道 < N（默认 2）"——宁可不动，不可把路由降到无路可走；
2. **步长与频率**：单次调整幅度由 step_ratio 限定；cooldown 强制间隔；每渠道每天最多 M（默认 6）次自动动作；
3. **人工优先**：快照比对发现渠道权重/优先级被人工改动过（非本系统命令所致的变化）→ 该渠道进入 24h 静默期，人工意图优先；
4. **只碰启用渠道**；实例 Agent 离线时不产生任何动作；
5. **可回滚**：每次执行记录 before/after（快照佐证），Web 一键"恢复原值"（同样走命令链路）；
6. **总开关联动**：`CT_NEW_API_CONTROL_ENABLED=false` 的 Agent 端天然拒绝执行——双端都要显式打开才真正生效。

## 5. 数据模型（新表 tuning_recommendations）

```
id, instance_id, channel_id, created_at,
rule (degrade|severe|recover), evidence_json,     -- 触发时的窗口指标快照（错误率/样本数/P95/连续周期数）
current_weight, current_priority, proposed_weight, proposed_priority,
mode_at_creation (observe|confirm|auto),
status (recorded|pending|adopted|auto_executed|dismissed|expired|rolled_back),
command_id (采纳/自动执行时关联渠道命令，审计链路复用),
outcome_json, outcome_at                          -- 30 分钟后回填：后续窗口错误率/P95 走势
```

**命中率定义**（回答"开自动合不合理"的量化口径）：
- 降级建议命中 = 建议后 30 分钟该渠道错误率仍 ≥ 阈值（说明"当时降是对的"）；
- 恢复建议命中 = 建议后 30 分钟错误率仍 ≤ 恢复阈值；
- Web 报表按最近 7/30 天汇总：建议总数、命中率、若为 confirm/auto 的采纳率与执行后改善率。**经验判据写进文档：观察期命中率持续 ≥85% 且无"最小可用集"险情，才建议切 auto。**

## 6. 执行链路（全部复用既有设施，零新通道）

```
评估引擎（Server 侧新 runner，每评估周期跑一次）
  → 建议落库
  → confirm 模式：Web 采纳 → POST channels/{id}/commands（confirm=true, created_by=操作人）
  → auto 模式：引擎直接建命令（created_by=auto-tuner）
  → 既有链路：心跳认领 → Agent channel.update → 结果回传 → operation_audits
```

审计、命令过期、实例匹配校验、Agent 端幂等全部免费继承——这正是 M1-B4 把命令闭环做扎实的回报。

## 7. Web 界面（调权中心，新页）

1. **模式与策略**：每实例模式三选一开关 + 策略表单（含护栏可配项），修改留痕；
2. **建议流水**：时间线式列表——证据快照（"错误率 18%，46 样本，连续 2 周期"）、当前值→建议值、状态、**事后走势徽标**（命中 ✓ / 未命中 ✗）；confirm 模式下带"采纳/忽略"按钮（采纳弹既有的危险操作确认框）；
3. **命中率报表**：7/30 天汇总卡片 + "切换 auto 的建议判据"提示；
4. **自动执行记录**：before/after、关联命令与审计、一键回滚按钮。

## 8. 批次拆分（v2.0 发布后启动）

| 批次 | 内容 | 风险 |
| --- | --- | --- |
| v2.1-B1 | 评估引擎 + 策略存储/API + 建议记录 + 事后回填（**仅 observe**） | 零（不产生任何动作） |
| v2.1-B2 | Web 调权中心（配置/流水/报表）+ confirm 模式采纳链路 | 低（每次动作人工确认） |
| v2.1-B3 | auto 模式 + 全部护栏 + 回滚 | 受控（数据达标才建议开启） |

**上线路径**：v2.1 全量部署后先跑 observe ≥2 周 → 看命中率报表 → 达标切 confirm 用一阵（体感核对）→ 再切 auto。每一步都可退回。

## 9. 已知边界

- 评估基于"已完成请求"，与告警一样存在观察滞后（约 1 个评估窗口）；探测数据并入后可加快劣化发现；
- new-api 侧权重语义（权重与优先级如何参与其负载均衡）以固定版本的实际行为为准，v2.1-B1 前需在验证环境实测确认一次（写入 evidence 文档）；
- 多实例各自独立评估互不影响；同一上游供应商在两实例各有渠道时不做跨实例联动（记为后续可能项）。
